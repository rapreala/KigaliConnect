## 1. Application Overview

KigaliConnect is a Flutter mobile application that serves as a real-time city directory for Kigali, Rwanda. It enables authenticated users to browse, search, and navigate to essential public services and leisure locations. Authenticated users can also contribute new listings and manage their own entries.

## 2. Firestore Schema Design

Two top-level collections were chosen to keep data access simple and avoid deep nesting.

### `users/{uid}`

Stores per-user profile and preferences. The document ID is the Firebase Auth UID, which enforces a strict ownership relationship in security rules.

| Field | Type | Purpose |
|---|---|---|
| uid | String | Mirror of the document ID |
| email | String | Display only (auth source of truth is Firebase Auth) |
| displayName | String | Shown in Settings avatar and listings attribution |
| createdAt | Timestamp | Account creation date |
| notificationsEnabled | Bool | User preference, toggled in Settings |

### `listings/{listingId}`

Stores all public place listings. The document ID is auto-generated by Firestore on creation.

| Field | Type | Purpose |
|---|---|---|
| id | String | Mirror of document ID for client-side filtering |
| name | String | Display name of the place |
| category | String | Enum name (e.g. `hospital`, `park`) |
| address | String | Human-readable street address |
| contactNumber | String | Phone number, validated 7‚Äì15 digits |
| description | String | Up to 500 characters |
| latitude | Number | Decimal degrees, -90 to 90 |
| longitude | Number | Decimal degrees, -180 to 180 |
| createdBy | String | UID of the creator ‚Äî used for ownership checks |
| createdAt | Timestamp | Creation time |
| updatedAt | Timestamp | Last modification time |

**Security Rules:**
- Any authenticated user can read all listings (public directory).
- A user can only create a listing where `createdBy == request.auth.uid`.
- A user can only update or delete a listing where `resource.data.createdBy == request.auth.uid`.

## 3. Listing Model Design

The `Listing` domain class is a pure Dart immutable value object with no Flutter or Firebase imports. This enforces clean architecture ‚Äî the domain layer has zero external dependencies.

Key design decisions:

- **`copyWith()`** ‚Äî enables immutable updates throughout the BLoC layer without mutation.
- **`==` / `hashCode`** ‚Äî based on all business fields so Equatable-style state comparisons work correctly in `ListingsLoaded.props`.
- **`fromJson` / `toJson`** ‚Äî the only layer that touches `cloud_firestore.Timestamp`. This keeps serialisation co-located with the model and out of the BLoC.
- **`category` as enum** ‚Äî `PlaceCategory` is serialised as its `.name` string (e.g. `"hospital"`), making Firestore documents human-readable and avoiding fragile integer indexes.

## 4. BLoC Architecture & Data Flow

The app uses **flutter_bloc 8.x** with a strict Events ‚Üí BLoC ‚Üí States pattern.

### ListingsBloc

The central piece of state management for the directory feature.

**Key design decisions:**
- **Single stream, client-side filter**: always stream all listings from Firestore and apply category/search filters in-memory. This makes category switching instant and avoids multiple Firestore stream subscriptions.
- **`allListings` field**: the unfiltered master list is exposed on `ListingsLoaded` so `MapViewScreen` always shows every marker regardless of what filter is active on the Places tab.
- **Stream-driven create**: rather than prepending to `_allListings` (which caused duplicates due to concurrent BLoC event processing), the create handler awaits the write and lets the Firestore local cache fire the stream ‚Äî the listing appears in under 100ms with no perceptible delay. Update and delete still mutate `_allListings` directly since replacing/removing by ID is idempotent.
- **Auto-resubscribe on error**: `_ListingsStreamErrored` is an internal event that calls `_subscribe()` again, keeping the real-time feed alive after a transient network error.

### AuthBloc

Wraps `FirebaseAuthRepository` and handles the full authentication lifecycle. It subscribes to `authStateChanges` so the UI reacts automatically when the Firebase token changes (e.g. email verified externally, account deleted).

### ThemeCubit

A lightweight `Cubit<ThemeMode>` backed by `SharedPreferences`. `loadSavedTheme()` is called before `runApp()` to prevent a flash of the wrong theme on startup.

## 5. Navigation Architecture

`AppShell` uses an `IndexedStack` with a `BottomNavigationBar` for four tabs: **Places**, **Map**, **My Listings**, **Settings**. `IndexedStack` preserves the scroll position and BLoC state of each tab when switching ‚Äî no rebuilds, no lost pagination.

A single `ListingsBloc` is provided above `AppShell` so all four tabs share the same real-time data stream. Creating a listing in the Places tab immediately shows the new marker on the Map tab with no extra round-trip.

## 6. Design Trade-offs

| Decision | Alternative Considered | Rationale |
|---|---|---|
| Client-side category filter | Server-side `where()` per category | Instant switching, no extra stream subscriptions, handles "empty category" UX gracefully |
| Single `ListingsBloc` above shell | Per-screen BLoC instances | Shared real-time state; map and directory always in sync |
| `IndexedStack` navigation | `Navigator` push/pop per tab | Preserves scroll position and avoids re-subscribing to Firestore on tab switch |
| Auto-geocoding on address input | Manual lat/lng entry only | Reduces friction for listing creation; falls back gracefully if geocoding fails |
| `allListings` + client filter | `filteredListings` only | Map needs unfiltered data; merging both into one state object avoids a second stream |
| Immutable domain models | Mutable models | Safe for BLoC equality checks and `props` comparison; prevents accidental mutation |
